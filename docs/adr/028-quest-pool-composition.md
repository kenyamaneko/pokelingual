# ADR-028: 出題プールの合成ロジックと静的ドメイン定数

## ステータス

Accepted

## 結論

出題対象プールは「世代フィルタ ∩ 除外（ユーザー設定 ∪ 開発者除外） ∩ （場所のタイプ ∪ 2段抽選で当たった場合の幻・伝説プール） ∩ `getServableIDs()`」の積集合として、サービス層（`quest-service`）が一元的に合成する。図鑑番号の上限（`MAX_POKEMON_ID` / mock の固定リスト）はプール生成時には適用せず、抽選時に `getServableIDs()` と突き合わせて一度だけ適用する。場所→タイプの対応（`domain/location.ts`）と幻・伝説の図鑑番号集合（`domain/legendary.ts`）は DB でなくコード上の静的定数として持つ。

## 背景・課題

出題プールに関わる条件（世代・除外・場所タイプ・幻伝説・上限）が増えるにつれ、どこで何を適用するかがサービス層とアダプタ層に分散しがちだった。

図鑑番号の上限は実際にはデータソース（PokeAPI または mock の固定リスト）が提供できる範囲そのものであり、プール生成側（世代境界の計算）にも上限を持たせると二重管理になる。実際に、プール生成関数から上限のクランプ処理を削除し `getServableIDs()` に一元化するリファクタが行われた（データソース側の上限が変わったときにプール生成側の追従漏れが起きうるため）。

場所→タイプ対応と幻・伝説の図鑑番号は、出題のたびに参照される静的な参照データ（ゲームの世界設定）であり、DB に置くと出題ごとに読み取りのレイテンシと課金が乗る。

## 詳細

- 抽選はサービス層（`quest-service`）が担う。アダプタ層（`PokemonClient`）は抽選ロジックを持たず、データ提供のみを担当する（`getServableIDs()`：データソースが提供できる図鑑番号、`getIDsByType(type)`：real は PokeAPI `/type/{name}` をキャッシュ、mock は固定リストをフィルタ）。
- 2段抽選：まず乱数上位 1%（`LEGENDARY_ENCOUNTER_RATE`）で幻・伝説を判定する。当たれば場所を無視して幻・伝説プール（`domain/legendary.ts` の固定集合）から、外れれば選んだ場所のタイプを持つポケモンから抽選する。幻・伝説プールが選択世代で空ならフォールバックして場所抽選する。
- 世代・除外との合成は AND：場所・幻伝説どちらのプールも、世代フィルタ・除外（`buildQuestPoolIDs`）および `getServableIDs()` と交差した上で抽選する。
- mock の決定性：`MockRandomSource` は 0 を返すため 1% 判定に当たらず（`>= 1 - RATE` で判定）、場所抽選が決定的に働く。
- プールが空（最低 1 世代必須・除外上限で通常防ぐが、設定次第で起こりうる）なら `EmptyQuestPoolError` → 409 で「設定を見直して」と案内する。

## 不採用案

- **プール生成時にも上限をクランプする**：データソース側の上限と二重管理になり、片方だけ更新されると齟齬が生じる。生成側は世代境界だけを表し、上限はデータソースに一元化した。
- **場所・幻伝説データを DB で管理する**：実行時に変化しない静的参照データに DB を使うと、出題のたびに読み取りのレイテンシと課金が乗る。内容変更にコード変更とデプロイが要る点はトレードオフとして残るが、運用者が画面から編集する要件が出たら設定ファイル化・DB 化を再検討する。
