# ADR-028: 出題プールの合成ロジックと静的ドメイン定数

## ステータス

Accepted

## 結論

出題対象プールは、世代フィルタ・除外設定（ユーザー設定と開発者除外の和）・場所条件（場所のタイプ、または2段抽選で幻・伝説に当たった場合の幻伝説プール）・データ取得層が提供できる範囲、の積集合として、サービス層が一元的に合成する。図鑑番号の上限は、抽選時にデータ取得層の提供範囲と突き合わせて一度だけ適用し、上限管理をデータ取得層に一元化する。場所→タイプの対応と幻・伝説の図鑑番号集合は、出題のたびに参照する静的な参照データであり、読み取りのレイテンシと課金を避けるためコード上の定数として持つ。

## 背景・課題

出題プールに関わる条件（世代・除外・場所タイプ・幻伝説・上限）が増えるにつれ、どこで何を適用するかがサービス層とアダプタ層に分散しがちだった。

図鑑番号の上限は実際にはデータソース（PokeAPI または mock の固定リスト）が提供できる範囲そのものであり、プール生成側（世代境界の計算）にも上限を持たせると二重管理になる。実際に、上限のクランプ処理をプール生成側から削除しデータ取得層に一元化するリファクタが行われた（データソース側の上限が変わったときにプール生成側の追従漏れが起きうるため）。

2段抽選は、まず乱数上位 1% で幻・伝説を判定し、当たれば場所を無視して幻・伝説プールから、外れれば選んだ場所のタイプを持つポケモンから抽選する、という優先順位で行う。幻・伝説プールが選択世代で空ならフォールバックして場所抽選する。

## Amendment: 2026-07-18 出現確率を運用時に設定可能にする

幻・伝説の出現確率 (1%) を env 経由で運用時に変更できる設定値にした。既定値は変えていない。

## 不採用案

- **プール生成時にも上限をクランプする**：データソース側の上限と二重管理になり、片方だけ更新されると齟齬が生じる。生成側は世代境界だけを表し、上限はデータソースに一元化した。
- **場所・幻伝説データを DB で管理する**：実行時に変化しない静的参照データに DB を使うと、出題のたびに読み取りのレイテンシと課金が乗る。内容変更にコード変更とデプロイが要る点はトレードオフとして残るが、運用者が画面から編集する要件が出たら設定ファイル化・DB 化を再検討する。
