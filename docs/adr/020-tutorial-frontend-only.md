# ADR-020: 初回チュートリアルを frontend 完結で実装する

## ステータス

Accepted

## 結論

チュートリアル (BDR-006) の判定・進行は、backend のクエスト API (PokeAPI 連携・LLM 採点・確率計算) を一切経由せず、frontend のローカル state だけで完結させる。登場ポケモンの情報・正答文字列は frontend にハードコードする。完了の永続化のみ、新規 backend エンドポイント経由で `users/{uid}` ルートドキュメントの `tutorial_completed` フィールドを更新する。`users/{uid}` はこれまでフィールドを持つドキュメントとして存在しなかったため、これを対象とする新規 Repository を追加する。

## 背景・課題

本番のクイズ判定 (英文当て・名前当て・採点・捕獲) は一貫して backend が担ってきた (`quest-service.ts`、ADR-001 の ports/adapter 設計)。チュートリアルも同じ経路を再利用するか、専用の簡易経路にするかの選択が必要だった。チュートリアルはモーダルで正答をそのまま提示するため判定を秘匿する理由がなく、frontend で完結させても実害がない。往復のネットワーク待ちを避けられる利点もある。

## 制約

- backend が Firestore への唯一のアクセス経路であるという既存原則 (`rules/principles.md`) は維持する。完了フラグの書き込みは新規エンドポイント経由で backend が行う。
- 契約型は `shared/api-types` を SSoT とする既存原則 (ADR-017) に従い、完了フラグ更新のリクエスト/レスポンス型をそこに追加する。

## 詳細

- 新規 backend エンドポイント (パス名は実装時に決定) を追加し、認証済みユーザー自身の `tutorial_completed` を `true` に更新する。
- `users/{uid}` ルートドキュメントを対象とする新規 Repository を追加する。
- チュートリアル中の判定 (キーワード一致・名前一致) は frontend 内のロジックとして実装し、backend の採点・判定ロジックとは共有しない (チュートリアル専用の固定判定のため)。

## 不採用案

既存の `/quest/new`, `/quest/score`, `/quest/guess`, `/quest/capture` に tutorial モード分岐を追加する案は、本番ロジックへの分岐追加とテスト影響が大きく、チュートリアルが完全固定シナリオである以上 PokeAPI・LLM・確率計算を経由するメリットがないため不採用とした。
