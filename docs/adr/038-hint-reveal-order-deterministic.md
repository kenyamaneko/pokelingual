# ADR-038: ヒントの開示順序を決定的にする

## ステータス

Accepted

## 結論

名前当てのヒントは、1回目に必ずタイプ、2回目に技を開示する決定的な順序で実装する。開示順をランダムにする案は採らない。

## 背景・課題

Issue #177で技ヒントを追加するにあたり、当初は「1回目にタイプ・技のどちらを開示するかをサーバがランダムに決め、2回目に残りを開示する」対称なランダム順を検討していた。

pokelingualはbackendが直近1つ前のfrontendと常に後方互換を保つ方針を採る (ADR-026)。デプロイはbackend→frontendの順で行われ、かつユーザーが既にブラウザで開いている旧frontendのSPAはページを再読み込みするまで旧のまま新backendを呼び続けるため、この互換窓は実運用でも起こりうる。既存の名前当て画面 (`frontend/src/components/quest/NameGuess.tsx`) は、ヒント結果を受け取ると以後ヒントボタンを表示しない実装になっており、旧frontendは1クエストにつき生涯1回しかヒントを要求できない。またヒント結果からタイプの案内文を組み立てる処理は、レスポンスの `types` を無条件で参照する。

ランダム順を採ると、旧frontendが送る唯一のヒント要求で技が先に選ばれた場合、レスポンスに `types` が含まれずこの処理が例外を投げる。技が先に選ばれる確率は五分五分のため、看過できない頻度で起こりうる。

開示順を「1回目は必ずタイプ」に固定すれば、旧frontendが受け取るレスポンスは常に `types` を含む形になり、後方互換を壊さずに1PR・1リリースのまま技ヒントを追加できる。

## 不採用案

- ランダム順を保ったまま、backendを新旧両応答対応にするリリース→frontendを新契約に切り替えるリリース→backendから旧対応を外すリリースの3リリースに分割する案。ADR-026が定める分割の型どおりだが、リリース数が増え実装・検証の手間が大きい。
- ランダム順を保ったまま、frontendからのリクエストに新契約を理解している旨のsignal (例: バージョン付きのリクエストパラメータ) を付け、backendがsignalの有無で応答を出し分ける案。1リリースで両立できるが、version negotiationの実装が加わり、機能の単純さに見合わない複雑さになる。

いずれも、開示順のランダム性という体験上の効果に対して実装コストが見合わないと判断し、順序を決定的にする案を採用した。
