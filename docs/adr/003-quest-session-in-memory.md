# ADR-003: クエストセッションのインメモリ管理

## ステータス

採用済み

## コンテキスト

クエストは4段階のフロー（new → score → guess-name → capture）で構成される。
各ステップ間でポケモン情報やスコアを保持する必要がある。

選択肢:
1. Firestore に保存 — 永続的だがレイテンシが高い
2. インメモリ (`sync.Map`) — 高速だがサーバー再起動で消失

## 決定

`sync.Map` によるインメモリ管理を採用。

## 理由

- クエストは数分で完了する短いフロー。永続化の必要性が低い
- Firestore 往復を省くことでレイテンシを大幅に削減
- Cloud Run のインスタンスが1つであるため、ユーザー間のセッション衝突は起きない
- 万が一セッションが消えても「新しいクエストを始める」だけで復帰可能

## トレードオフ

- Cloud Run インスタンスのスケールアウト時、同一ユーザーが異なるインスタンスに振られるとセッションが見つからない
- 現状のユーザー数（1-2人）では問題にならない。スケールが必要になった時点で Redis 等への移行を検討
