# ADR-025: CI/CD パイプライン再構成 (ランナーテストを主ゲート、デプロイ後は dev スモークのみ)

## ステータス

Accepted (ADR-016 のデプロイ後統合テスト＋自動ロールバックを置き換える)

## 結論

PR・main マージ・タグの各トリガーで、共有 CI (lint・バックエンド・フロントエンド・E2E) をランナー上で走らせ、これを品質の主ゲートとする。main マージでは CI が通ったときだけ dev へデプロイし、デプロイ後は**検出専用スモーク**を 1 本 (ヘルス＋認証付き read) だけ実行する。自動ロールバックはしない。タグ push では**テストを走らせず**、同一コミットの backend を prod 用に再ビルドしてデプロイする。フロントエンドも環境値をビルド時に埋め込むため prod 用に再ビルドし、表示バージョンはタグになる。

これにより、タグ時のテスト再実行という二重実行がなくなり、dev/prod のワークフローが分離されて各トリガーの流れが読みやすくなり、ランナーテストでは捉えられない実環境固有の壊れを最小限のコストで検知できる。

## 背景・課題

GitHub Flow (main 1 本、ADR-019) への移行に伴い、環境反映を「main マージ → dev、タグ → prod」に整理した。

旧構成 (ADR-016) は dev デプロイ後に実 backend へ統合テストを走らせ、失敗時に前リビジョンへ自動ロールバックしていた。しかし CI が通るまで dev にデプロイしない設計では、「テストが失敗した状態でデプロイされる」状況が構造上発生しない。ロールバックの契機が消えるため、デプロイ後にフルの統合テストとロールバックを持つ意味が薄れた。

一方でタグ prod デプロイは、dev で既に検証済みの同一コミットに対してテストを再実行しており、同じコミットへの二重実行で新しい情報を生んでいなかった。とくに E2E は所要時間が長く、この再実行が最大の無駄だった。

Vite は API URL・バージョン・環境名をビルド時にバンドルへ埋め込むため、フロントエンドは環境ごとに再ビルドが必要になる。dev と prod は別々の Google Cloud プロジェクトで、Artifact Registry も別々である。Cloud Run の `--allow-unauthenticated` は反映に時間がかかることがあり、その間はヘルスも IAM レベルで拒否される (ADR-015)。

## 不採用案

- **デプロイ後検証を全廃しランナーテストのみにする**: ランナーテストはエミュレータ・モックに対して走るため、実環境の secret・env・IAM の壊れを構造的に検知できない。それを最小限捉えるため、ヘルス＋read の薄いスモークを残した。
- **単一共有レジストリでバイト同一を昇格する**: プロジェクトをまたぐ IAM と prod↔dev のレジストリ結合が必要で、削減できるのは数分の再ビルドのみ。リリース頻度が低いため、プロジェクト分離を保ち prod で再ビルドする方式を採った。
- **タグ push でテストも再実行する**: 同一コミットへの二重実行で新しい情報を生まないため、タグ時はテストを走らせない。
- **統合テスト＋自動ロールバックを維持する (ADR-016)**: CI が通るまでデプロイしない設計ではロールバックの契機が発生せず、ソロ運用には過剰なため置き換えた。

## Amendment: 2026-07-12 dev スモークへのクエスト経路追加と prod スモークの新設

dev のスモークに、フィクスチャユーザでのクエスト 1 周分の実行を追加する。実 Gemini 採点とレート制限カウンタ・図鑑への Firestore 書き込みの経路は、既存のヘルス＋read だけでは検出できないため。書き込みを避けるという当初の制約から外れるが、フィクスチャユーザのカウンタ消費と図鑑へのデータ蓄積は許容し、後始末はしない。

prod にも同じ設計のスモーク (ヘルス＋認証付き read のみ) をデプロイ後に追加する。prod の認証は公開モードに切り替えたため許可リスト常設の手順は不要だが、認証付き read を叩くためのフィクスチャユーザ自体は dev と同様に必要になる。prod のレート制限枠は実ユーザーと共有するため、Gemini 経路は prod スモークには含めない。ロールバックはしない (通知のみ) という設計は dev と変わらない。
