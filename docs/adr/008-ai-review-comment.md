# ADR-008: 翻訳スコアリングに AI レビューを追加

## ステータス

採用済み（更新: comment → review にリネーム、表示位置を QuestPage に移動）

## コンテキスト

翻訳スコアリングでは 0-100 の数値スコアのみを返していた。
ユーザーが「なぜこの点数なのか」を理解するための手がかりがなく、学習体験としてフィードバックが不十分だった。

また、スコアラベル（「効果は ばつぐんだ！」等）と AI フィードバックの両方を「コメント」と呼ぶと紛らわしいため、AI フィードバックは `review` に統一した。

## 決定

Gemini API のプロンプトに `review` フィールドを追加し、日本語レビューを返すようにする。

### 変更箇所

- **ScoreResult**（`model/quest.go`）: `Review string` フィールド追加
- **ScoreResponse**（`service/quest_service.go`）: `Review string` フィールド追加、サービス層でパススルー
- **GeminiService**（`service/gemini_service.go`）: プロンプトに review 要求を追加
- **devmock**: スコア範囲に応じた固定レビューを返す
- **フロントエンド**: `QuestPage` で「きみの 翻訳」の下にレビューを表示

### レビューのガイドライン

- 日本語で 2-3 文（100文字以内）
- PokeAPI の日本語説明文スタイル（簡単な漢字 + スペース区切り）
- 以下の内容を適宜含む:
  - ユーザーの翻訳の良い点（語彙選択、文法など）
  - 改善点（ニュアンスの欠落、不自然な表現、誤訳）
  - 難しい英単語・フレーズの解説（ユーザーが苦戦していそうな場合）
- 励ましつつ正直に — ポケモン博士のような口調で
- 例: `「friskily」は 元気よく 跳ね回る という 意味だよ。前半は よく 訳せているが 後半の ニュアンスが 少し 惜しい。`

### 命名規則

| 概念 | コード上の名前 | 表示場所 |
|------|---------------|---------|
| スコアラベル（「効果は ばつぐんだ！」等） | `getScoreLabel()` | ScoreDisplay |
| AI フィードバック | `review` | QuestPage（「きみの 翻訳」の下） |

## 結果

- ユーザーは点数の理由を理解し、次に活かせる具体的なフィードバックを得られる
- 難しい単語の解説により、英語学習の効果が向上
- Gemini API へのリクエスト数は変わらない（既存のスコアリングリクエストにレビューを追加しただけ）
- devmock でもスコア範囲に応じたレビューを返すため、ローカル開発でも体験を確認可能
- スコアラベルと AI レビューの命名が明確に分離され、コードの可読性が向上
