# ADR-011: AI 呼び出しの日次レートリミット

## ステータス

採用済み

## コンテキスト

アプリを SNS で公開すると、Gemini API への呼び出しが従量課金で青天井になる。
1人の悪意あるユーザーが連打すれば月予算（5,000円）を1日で使い切る可能性もある。
個人ポートフォリオなので「予算が読める」「事故っても財布が守られる」状態を保ちたい。

選択肢:

1. アプリ層で per-user の日次上限のみ
2. アプリ層で per-user + 全体（global）の日次上限の二段
3. GCP Billing Budget で自動停止（Pub/Sub + Cloud Function）
4. なし（Billing アラートメールのみ）

## 決定

**選択肢 2 を採用。アプリ層で per-user 30回/日 + global 1,500回/日の二段制限。**

カウント対象は `/api/quest/score` と `/api/quest/chat`（Gemini を呼ぶエンドポイント）のみ。
`/api/quest/new`（PokeAPI のみ、Gemini 不使用）は対象外。

数え方は「AI 呼び出し1回」を1単位とする。クエスト単位ではなくチャット往復もカウント。
コスト直結で、ユーザーへの表示も「のこり 30/30 回」と単純化できる。

### 上限値の根拠

| 項目 | 値 | 根拠 |
|---|---|---|
| 月予算 | 5,000円 | 個人ポートフォリオの許容額 |
| AI 1呼び出し平均コスト | 約 0.11円 | thinking 無効化後、入力 800-1200tok + 出力 150-200tok |
| 月間処理可能呼び出し | 約 45,000回 | 5000 ÷ 0.11 |
| 1日あたり global 上限 | 1,500回 | 45000 ÷ 30 |
| 1ユーザー1日 | 30回 | 体験として「もう少し遊びたい」で終わる量。50人/日の同時収容 |

最大シナリオでも `1,500 × 30 × 0.11 = 4,950円` で月予算に収まる。

### リセット境界

JST 0:00 固定。ユーザーの現地時間にすると 23:59 + 0:00 で枠を2倍使える抜け道ができるため。

### グローバル上限優先

トランザクション内で **global を先に判定**。これにより 1人の濫用者が global を詰めた場合でも、
他のユーザーには `kind: "global"` を返せる（「あなたが悪い」ではなく「今日は混雑」と伝えられる）。

### Firestore 構造

```
users/{uid}/daily_usage/{YYYY-MM-DD}  → { count, updated_at }
system/daily_usage/{YYYY-MM-DD}       → { count, updated_at }
```

日付キーにすることで翌日0時に自動的に新ドキュメントへ切り替わり、リセット処理不要。
古いドキュメントは Firestore TTL で削除（運用上必要になったら設定）。

### 自動停止を採用しない理由

選択肢 3（Billing Budget → Cloud Function 自動停止）は実装可能だが見送り:

- Billing 通知は数時間〜半日遅延。発火する頃には予算を超えている
- アプリ層レートリミットが「実質的な上限装置」として既に機能している
- Cloud Function + Pub/Sub + IAM の追加複雑度に見合うメリットがない

Billing Budget はメール通知のみ（選択肢 4）で「想定外コストの早期検知」用途として残す。

## 結果

- 月予算 5,000円が（バグがない限り）絶対に超えない設計
- 50人/日が普通に遊べる収容力
- 上限到達時は博士口調モーダルでゲーム演出として見せる（エラーではない）
- ヘッダーに「のこり N/30 回」を常時表示してユーザーが残量を把握できる
- Billing アラートメールで二重防御
