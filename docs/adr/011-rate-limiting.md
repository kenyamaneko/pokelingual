# ADR-011: AI 呼び出しの日次レートリミット

## ステータス

Accepted

## 結論

月予算 5,000 円を超えないようにするため、AI 呼び出しにアプリ層の二段日次上限（per-user 30 回/日 + global 1,500 回/日）を設ける。カウント対象は Gemini を呼ぶ `/api/quest/score` と `/api/quest/chat` のみ。バグがない限り予算を超えず、50 人/日が普通に遊べる収容力を保ちつつ、上限到達は博士口調モーダルでゲーム演出として見せる。

## 背景・課題

アプリを SNS で公開すると、Gemini API への呼び出しが従量課金で青天井になる。1 人の悪意あるユーザーが連打すれば月予算（5,000 円）を 1 日で使い切る可能性もある。個人ポートフォリオなので「予算が読める」「事故っても財布が守られる」状態を保ちたい。

## 詳細

数え方は「AI 呼び出し 1 回」を 1 単位とし、クエスト単位ではなくチャット往復もカウントする。コスト直結で、ユーザーへの表示も「のこり 30/30 回」と単純化できる。`/api/quest/new`（PokeAPI のみ、Gemini 不使用）は対象外。

### 上限値の根拠

| 項目 | 値 | 根拠 |
|---|---|---|
| 月予算 | 5,000円 | 個人ポートフォリオの許容額 |
| AI 1呼び出し平均コスト | 約 0.11円 | thinking 無効化後、入力 800-1200tok + 出力 150-200tok |
| 月間処理可能呼び出し | 約 45,000回 | 5000 ÷ 0.11 |
| 1日あたり global 上限 | 1,500回 | 45000 ÷ 30 |
| 1ユーザー1日 | 30回 | 体験として「もう少し遊びたい」で終わる量。50人/日の同時収容 |

最大シナリオでも `1,500 × 30 × 0.11 = 4,950円` で月予算に収まる。

### リセット境界

JST 0:00 固定。ユーザーの現地時間にすると 23:59 と 0:00 で枠を 2 倍使える抜け道ができるため。

### グローバル上限優先

トランザクション内で **global を先に判定**する。1 人の濫用者が global を詰めた場合でも、他のユーザーには `kind: "global"` を返せる（「あなたが悪い」ではなく「今日は混雑」と伝えられる）。

### Firestore 構造

```
users/{uid}/daily_usage/{YYYY-MM-DD}  → { count, updated_at }
system/daily_usage/{YYYY-MM-DD}       → { count, updated_at }
```

日付キーにすることで翌日 0 時に自動的に新ドキュメントへ切り替わり、リセット処理は不要。古いドキュメントは Firestore TTL で削除する（運用上必要になったら設定）。

## 不採用案

- **per-user の日次上限のみ**：1 人あたりは抑えられても、多数のユーザーが同時に叩くと global で予算を超える。
- **Google Cloud Billing Budget で自動停止（Pub/Sub + Cloud Function）**：Billing 通知は数時間〜半日遅延し、発火する頃には予算を超えている。アプリ層レートリミットが実質的な上限装置として既に機能しており、Cloud Function + Pub/Sub + IAM の追加複雑度に見合わない。
- **上限なし（Billing アラートメールのみ）**：事故時に財布が守られない。ただし Billing アラートメールは「想定外コストの早期検知」として二重防御に残す。

## Amendment: 2026-05-30 global カウンタのパス修正

設計時の global カウンタのパス `system/daily_usage/{YYYY-MM-DD}` は、実装で `system/global/daily_usage/{YYYY-MM-DD}` に修正した。Firestore のドキュメントパスは collection/doc の交互で偶数階層が必須のため、3 階層の元案は無効パスだった（Firestore Emulator テストで顕在化）。意味的な変更はなく、global カウンタの永続化先として機能する。

## Amendment: 2026-07-05 博士チャット廃止に伴うカウント対象の縮小

博士チャット機能（`/api/quest/chat`）を廃止した（Issue #52）。チャットが採点と同じカウンタを消費するため「枠 = クエスト回数」の直感と実態がズレる、という UX 上の判断であり、コスト超過が理由ではない。これによりカウント対象は `/api/quest/score` のみとなった。二段日次上限の設計と上限値は変更しない。
