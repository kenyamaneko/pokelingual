# ADR-030: 初回チュートリアルを frontend 完結で実装する

## ステータス

Accepted

## 結論

チュートリアル（BDR-006）の判定・進行は、backend のクエスト API（PokeAPI 連携・LLM 採点・確率計算）を一切経由せず、frontend のローカル state だけで完結させる。登場ポケモンの情報・正答文字列は frontend にハードコードする。完了の永続化のみ、新規 backend エンドポイント経由でユーザーの完了フラグを更新する。

## 背景・課題

本番のクイズ判定（英文当て・名前当て・採点・捕獲）は一貫して backend が担ってきた（ADR-007 の ports/adapter 設計）。チュートリアルも同じ経路を再利用するか、専用の簡易経路にするかの選択が必要だった。チュートリアルはモーダルで正答をそのまま提示するため判定を秘匿する理由がなく、frontend で完結させても実害がない。往復のネットワーク待ちを避けられる利点もある。

backend が Firestore への唯一のアクセス経路であるという既存原則は維持し、完了フラグの書き込みは新規エンドポイント経由で backend が行う。契約型は `shared/api-types` を SSoT とする既存原則（ADR-027）に従う。

## 不採用案

既存の `/quest/new`, `/quest/score`, `/quest/guess`, `/quest/capture` に tutorial モード分岐を追加する案は、本番ロジックへの分岐追加とテスト影響が大きく、チュートリアルが完全固定シナリオである以上 PokeAPI・LLM・確率計算を経由するメリットがないため不採用とした。

## Amendment: 2026-07-16 本番 useQuest への注入駆動に変更

チュートリアルの進行を専用フック (useTutorialQuest) で並行実装していたのをやめ、本番の useQuest に固定 API (インメモリ実装) と固定出題を注入して駆動する形に変えた。並行実装が本番からドリフトし、捕獲演出 (ボールの揺れ) の欠落と名前当て結果のボールへの未反映という不具合が生じていたため、状態遷移と捕獲演出のコンポーネントを本番と共有してドリフトを止める。

この再利用は不採用案で退けた backend への tutorial 分岐追加とは別物で、backend は依然として一切呼ばない (frontend 完結は維持)。useQuest には tutorial 専用の条件分岐を入れず、汎用の差し替え口 (API 実装・初期出題・採点後の副作用) だけを追加したので、本番の経路は分岐のないまま保たれる。チュートリアルが backend を呼ばないことにより、捕獲は図鑑・実績に記録されない (BDR-006 2026-07-16 の記録方針と整合する)。
