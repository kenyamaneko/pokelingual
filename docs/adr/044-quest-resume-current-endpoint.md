# ADR-044: 進行中クエストの読み出し専用エンドポイントで再開を実現する

## ステータス

Accepted

## 結論

進行中セッションの読み出し専用エンドポイントを新設し、backend がセッションの状態から現在の段階を判定してフロントエンドに返す。フロントエンドは起動時にこのエンドポイントへ問い合わせ、結果に応じて画面の状態を復元するか、既存の新規開始フローに進む。

セッションへの追加項目は最小限に留め、ユーザの翻訳文と採点講評の2つだけを保持する。名前当ての残り試行回数やヒントの開示状況は、既存のセッション項目から都度導出する。

段階の判定は、ボール種別が確定しているかを最優先で見る。名前当てを採点前に済ませる経路があるため、得点の有無より先にボールの確定を見ないと、捕獲待機の段階を名前当ての段階と誤判定する。

セッションは説明文をポケモン名入りの生のまま保持しているため、再開時の応答を組み立てる際は伏せ字化を必ずやり直す。

## 背景・課題

BDR-012 の再開を実現するには、離脱した段階と、その段階の表示に必要な状態をどこかで再構成する必要がある。厳選封じが目的のため、判定はクライアントではなく backend が権威を持つ必要がある。

## 不採用案

- **フロントエンド側での状態保持**：ローカルストレージ等でのブラウザ側保持は、消去や別端末からのアクセスで容易に回避でき、リロードによる厳選封じという目的を達成できない。
- **セッションに再開用の状態を全て書き込む**：既存の項目から導出できる状態まで都度書き込むと、書き込み漏れによる不整合の芽が増える。導出できる範囲は保存せず、読み出し時に都度算出する方針にした。
