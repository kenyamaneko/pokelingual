# ADR-034: 初回チュートリアルを本番クエストドメインの別コンポジションで提供する

## ステータス

Accepted (Supersedes ADR-030)

## 結論

チュートリアルは、本番の `QuestService` / `QuestHandler` をドメイン無改変のまま再利用し、チュートリアル専用アダプタに差し替えたコンポジションを `/tutorial/quest/*` へ別マウントして提供する。差し替えるのは、ピカチュウ固定の PokemonClient・常に満点の LLMClient・決定的な RandomSource・記録しない UserPokemonRepository・既定設定の UserSettingsRepository。名前当ての判定・ボール選択・捕獲率は本番ロジックをそのまま通し、レート制限はチュートリアルには噛ませない。frontend は本番の QuestPage を再利用し、チュートリアル用 API クライアント・入力検証・案内 (吹き出し) を注入する。

## 背景・課題

ADR-030 はチュートリアルを frontend 完結とし、判定・データを frontend にハードコードしていた。その後、名前当ての結果によるボール分岐や捕獲演出を本番と揃えたい要求が生じ、frontend の固定実装が本番ロジックの再実装になってドリフトした。判定ロジックは backend の `QuestService` にのみあるため、本番と同じ挙動を保つには backend の判定を再利用する必要がある。

ports/adapter 設計 (ADR-007) により、`QuestService` はデータ源・採点・乱数・永続化をすべてポートで受け取る。チュートリアルはこれらを差し替えた別コンポジションとして表現でき、ドメインに分岐を持たせずに固定シナリオ (ピカチュウ・満点・必ず捕獲成功) を実現できる。捕獲を記録しない要件は、記録しないリポジトリを差し込むことでハンドラを変えずに満たせる。往復のネットワーク待ちより、本番ロジックとの一致とドリフト防止を優先する。

## 不採用案

- **frontend 完結 (ADR-030) の継続**：判定ロジックが backend にしかないため、frontend で本番と同じボール分岐・捕獲率を再現するには再実装が必要になり、二重管理でドリフトする。
- **本番エンドポイントに tutorial モード分岐を足す**：ドメイン・ハンドラに条件分岐が入り、本番ロジックとテストが複雑になる。コンポジションとルーティングで制御すれば分岐は不要になる。
