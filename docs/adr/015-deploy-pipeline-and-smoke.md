# ADR-015: CI/CD パイプライン再構成 (ランナーテストを主ゲート、デプロイ後は dev スモークのみ)

## ステータス

Accepted (ADR-006 のデプロイ後統合テスト＋自動ロールバックを置き換える)

## 結論

PR・main マージ・タグの各トリガーで、共有 CI (lint・バックエンド・フロントエンド・E2E) をランナー上で走らせ、これを品質の主ゲートとする。main マージでは CI が通ったときだけ dev へデプロイし、デプロイ後は**検出専用スモーク**を 1 本 (ヘルス＋認証付き read) だけ実行する。自動ロールバックはしない。タグ push では**テストを走らせず**、同一コミットの backend を prod 用に再ビルドしてデプロイする。フロントエンドも環境値をビルド時に埋め込むため prod 用に再ビルドし、表示バージョンはタグになる。

これにより、タグ時のテスト再実行という二重実行がなくなり、dev/prod のワークフローが分離されて各トリガーの流れが読みやすくなり、ランナーテストでは捉えられない実環境固有の壊れを最小限のコストで検知できる。

## 背景・課題

GitHub Flow (main 1 本、ADR-009) への移行に伴い、環境反映を「main マージ → dev、タグ → prod」に整理した。

旧構成 (ADR-006) は dev デプロイ後に実 backend へ統合テストを走らせ、失敗時に前リビジョンへ自動ロールバックしていた。しかし CI が通るまで dev にデプロイしない設計では、「テストが失敗した状態でデプロイされる」状況が構造上発生しない。ロールバックの契機が消えるため、デプロイ後にフルの統合テストとロールバックを持つ意味が薄れた。

一方でタグ prod デプロイは、dev で既に検証済みの同一コミットに対してテストを再実行しており、同じコミットへの二重実行で新しい情報を生んでいなかった。とくに E2E は所要時間が長く、この再実行が最大の無駄だった。

## 制約

- Vite は API URL・バージョン・環境名をビルド時にバンドルへ埋め込むため、フロントエンドは環境ごとに再ビルドが必要になる。
- dev と prod は別々の Google Cloud プロジェクト (`pokelingual-dev` / `pokelingual-prod`) で、Artifact Registry も別々。Terraform は両者へ同一モジュールを workspace 別に apply する対称構造をとる。dev でビルドした backend イメージは prod のレジストリに無いため、単一レジストリでのバイト同一昇格にはプロジェクトをまたぐ IAM と prod↔dev のレジストリ結合が要る。一方タグ (リリース) は頻度が低く、prod で同一コミットを再ビルドするコストは小さい。
- Cloud Run の `--allow-unauthenticated` は反映に時間がかかることがあり、その間はヘルスも IAM レベルで拒否される (ADR-005)。

## 詳細

ワークフローは 3 ファイルに分ける。

- `ci.yml`: 共有 CI。PR (`pull_request`) と `workflow_call` で起動し、lint・バックエンド・フロントエンド・E2E・Terraform fmt・テスト観点カタログ (ジョブ要約) を走らせる。
- `deploy-dev.yml`: `main` への push で起動。CI を再実行し、通れば dev へ backend/frontend をデプロイ、デプロイ後にスモークを走らせ、テスト観点カタログを GitHub Pages へ公開する。表示バージョンは `git describe --tags --always`。
- `deploy-prod.yml`: `v*` タグ push で起動。テストは走らせず、同一コミットを prod レジストリへ再ビルドして prod へデプロイし、frontend も prod 用に再ビルドしてデプロイする。表示バージョンはタグ。

スモーク (`scripts/smoke-dev.sh`) は、`GET /health` (起動確認) と、許可リストに常設したフィクスチャユーザでの `GET /api/usage` (実 Firebase Auth・Firestore・env/secret の配線確認) の 2 本のみを叩く。書き込みをしないため後始末は不要で、失敗しても通知のみとする。

テスト観点カタログはマージ時点の仕様を表す。テストが走るのは main マージの実行だけなので、公開もそのタイミングで行う。

## 不採用案

- **デプロイ後検証を全廃しランナーテストのみにする**: ランナーテストはエミュレータ・モックに対して走るため、実環境の secret・env・IAM の壊れを構造的に検知できない。それを最小限捉えるため、ヘルス＋read の薄いスモークを残した。
- **単一共有レジストリでバイト同一を昇格する**: プロジェクトをまたぐ IAM と prod↔dev のレジストリ結合が必要で、削減できるのは数分の再ビルドのみ。リリース頻度が低いため、プロジェクト分離を保ち prod で再ビルドする方式を採った。
- **タグ push でテストも再実行する**: 同一コミットへの二重実行で新しい情報を生まないため、タグ時はテストを走らせない。
- **統合テスト＋自動ロールバックを維持する (ADR-006)**: CI が通るまでデプロイしない設計ではロールバックの契機が発生せず、ソロ運用には過剰なため置き換えた。
