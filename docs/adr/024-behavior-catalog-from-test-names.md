# ADR-024: テスト名から振る舞いカタログを生成して GitHub Pages に公開する

## ステータス

Accepted

## 結論

テストを読めば仕様が把握できる状態を CI が保つため、テスト実行の結果からテスト名を集めた**振る舞いカタログ**を生成する。main への push ではカタログを GitHub Pages (<https://kenyamaneko.github.io/pokelingual/>) に公開し、PR では job summary に出力してレビューで振る舞いの増減を確認できるようにする。テストの命名規約（keyandnotes-rules testing.md「テストの命名」）がテスト名を仕様の一文として書くことを求めているため、テスト名の集約がそのまま仕様ドキュメントになる。

## 背景・課題

テストの命名規約により、各テストケース名は「〜のとき、〜すると、〜になる」の仕様の一文として書かれている。しかし各ケース名はテストコードの中に散在しており、仕様の全体像を掴むにはテストファイルを渡り歩く必要があった。テスト名を一覧に集約して人が読める形で公開する仕組みを CI に組み込む（Issue #54）。

private リポジトリでも GitHub Pages のサイト自体は誰でも閲覧できるが、カタログの中身はテスト名（振る舞いの記述）のみで秘匿情報を含まないため問題ない。CI コスト削減方針に従い、追加ジョブは軽量に保つ。

## 不採用案

- **テストソースの静的解析でテスト名を抽出する**：`it.each` のプレースホルダのように動的に組み立てるテスト名は、実行しないと確定しない。実行結果からの抽出なら確定済みの名前だけを扱える
- **カタログを docs/ にコミットする**：テストを変更するたびに生成物のコミットが必要になり、実装と生成物の二重管理になる。CI の成果物として都度生成する

## Amendment: 2026-07-12 テスト観点カタログへの改称と読者別カテゴリの追加

名称を**振る舞いカタログ**から**テスト観点カタログ**に改める。backend のテストを「API を使う人」向けと「backend 開発者」向けに分ける裁定を機に、frontend・E2E も含めた全セクションを次の 2 カテゴリに整理した。

- **外から見た振る舞い**：backend API の振る舞い、frontend、E2E。製品・API の利用者が読める言葉で書く
- **内部の挙動**：backend 内部部品の検証。backend 開発者が読める、部品の契約の言葉で書く

## Amendment: 2026-07-15 frontend の画面仕様・内部仕様への分割とセクションラベルの改称

frontend のうち `src/api/` のテスト (トークン付与・上限通知イベントの発火・診断ログ) は画面に対応物の無い内部配線を確かめており、画面仕様のセクションに混ざると読者が意味を取れない。そこで frontend を、画面から観測できる振る舞いの**frontend（画面仕様）**（「外から見た振る舞い」）と、内部配線の**frontend（内部仕様）**（「内部の挙動」）に分ける。「内部の挙動」は backend 内部部品の検証に加え、この frontend 内部配線の検証も含むようになる。

あわせて、backend・frontend を同じ二軸で読めるよう、各セクションのラベルをそのセクションが載せる仕様の種別に揃える。backend の 2 セクションを**backend（API仕様）**と**backend（内部仕様）**に改称する。

## Amendment: 2026-07-17 実装言語 (Python) の選定理由を不採用案に補記

不採用案に **TypeScript で実装する** を追加する。JUnit XML のパースは Python 標準ライブラリの `xml.etree.ElementTree` で完結するが、TypeScript では XML パーサーライブラリの追加依存が必要になる。依存を増やさずに実装できる Python を採る。

## Amendment: 2026-07-17 トップレベル describe 名の `[タグ]` による上位グルーピング

トップレベル describe 名の先頭に `[認証系]` のような `[タグ]` を付けられるようにする。カタログ生成時にタグを上位グループ見出しとして抽出し、タグを剥がした describe 名をその見出しの下に並べる。タグの無いトップレベル describe は「その他」にまとめる。同じタグは、複数のテストファイルに分かれていても 1 グループに集約する。目印をテストコード側（describe 名）に置くことで、describe のリネームやファイル移動と一緒に動き、別ファイルの保守を要らなくする。

### 不採用案

- **外部設定ファイルでグルーピングを指定する**：セクション → グループ → describe 名の対応と順序を別ファイルに書き、スクリプトが読んでグルーピングする案。テスト名を変えず順序も完全に制御できるが、describe をリネームすると設定がズレる (CI で検知はできるが別ファイルの保守が要る)。取りこぼしを防ぐには全 describe を設定ファイルへハードエラーで強制列挙する必要もある。タグ案はズレず・別ファイル不要・段階導入できるため、こちらを不採用とした。

## Amendment: 2026-07-18 タグの無いセクションでは「その他」でまとめない

セクション内にタグ付き describe が 1 件も無いときは「その他」でまとめない。まとめる相手がいないセクションまで機械的に 1 段深くすると、見出しが箇条書きに格下げされるだけで読みやすさの向上が無いため。
