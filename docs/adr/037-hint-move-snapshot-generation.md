# ADR-037: 覚える技ヒントを生成時に確定してスナップショットへ保存する

## ステータス

Accepted

## 結論

スナップショット生成時に、各ポケモンがレベルアップで覚える技から最大3つを選び、日本語名としてスナップショットへ保存する。実行時にPokeAPIや技の全リストを参照しない設計を維持する。

## 背景・課題

ADR-031により、pokelingualは実行時・生成時のいずれもpokeapi.coを呼ばず、非公開のCloud Storageスナップショットを唯一のデータソースとする方針を採る。Issue #177で技名をヒントに追加するには、この方針を保ったまま技データも生成時に解決し尽くす必要がある。

PokeAPIの技データはversion_group (作品) ごとに習得方法が異なる。その種がある作品の図鑑に登場しない場合、その作品のデータ自体を持たない (例: SpindaとSmeargleはsword-shieldのレベルアップ技データを持たない)。図鑑説明文の生成 (`backend/scripts/lib/flavor-text.ts`) では、優先順のバージョンリストを走査し、EN/JAの説明文が揃う最初のバージョンを採用するフォールバックを既に実装している。技の解決でも同型のフォールバックが要る。対象はversion_group単位のため、バージョン単位のリストとは別に、version_group単位の優先順リストを用意し、直近の作品から遡ってレベルアップ技を持つ最初のversion_groupを採用する。

技は多くのポケモンで共有されるため、日本語名の解決を種ごとに都度行うと同じ技を何度も読むことになる。生成スクリプトは全種の候補技を先に集約し重複排除したうえで、必要な技だけmove/{id}を1回ずつ読み、技slug→日本語名のマップを作ってから各種の変換に利用する。

## 不採用案

生成時に確定させず、スナップショットには対象作品の全レベルアップ技を保持し、ヒント要求時にランダムに3つを抽選する案を検討した。ADR-031の実行時方針そのものには反しないが、スナップショットのサイズが増えるうえ、抽選ロジックをbackend serviceに新たに持ち込む必要がある。データは生成時に確定させ実行時は読むだけ、という既存の設計(ADR-031)と一貫させるため不採用とした。

## Amendment: 2026-07-18 選出をクエスト開始時のランダム抽選に変更

PRレビューで、生成時に3技を確定する設計では同じ種に何度出会っても常に同じ3技しか開示されず、繰り返しの遭遇に新しい発見が無いという指摘を受けた。検討の結果、上記「不採用案」としていた案を採用に切り替える。

スナップショットには、各ポケモンがレベルアップで覚えうる技の候補を全件、日本語名として保存する (生成時の技候補解決・日本語名解決のロジックは本文の記述のまま変わらない)。ヒントとして見せる最大3件の選出は、生成時ではなくクエスト開始時にbackend serviceが乱数で行う。実行時にPokeAPIや技の全リストを参照しない設計 (ADR-031) は維持する。

既存のbackend serviceは出題ポケモンの抽選・幻/伝説抽選・複数バージョンがある説明文の選択を、クエスト開始時に注入済みの乱数源で行っている。技ヒントの選出もこれと同じ箇所・同じ乱数源で行うことで、抽選ロジックの置き場所を既存の設計と揃えた。本文の不採用理由に挙げた「抽選ロジックをbackend serviceに新たに持ち込む必要がある」というコストは、この既存の仕組みに1箇所加えるだけで済み、当初の見積もりほど大きくなかった。
