# ADR-022: ポケモン種別データを静的スナップショットから供給する

## ステータス

Proposed

## 結論

実行時に PokeAPI を呼ぶのをやめ、非公開の Cloud Storage バケットに置いた静的スナップショットからポケモンの種別データ（名前・説明文・タイプ・種族値など）を供給する。スナップショットは PokeAPI が公式に配布する静的データ（`PokeAPI/api-data`）から生成し、生成時も pokeapi.co を呼ばない。あわせて画像の配信元を jsDelivr に移す。

実行時・生成時ともに PokeAPI への呼び出しがゼロになる。コールドスタートのたびに走っていた再取得とレイテンシが消え、PokeAPI の障害が本番に波及しなくなる。画像配信も GitHub raw のレート制限から外れる。

## 背景・課題

現在のキャッシュは `PokeAPIClient` のインスタンス内メモリ（`Map`）だけで、プロセスの寿命を超えて残らない。

backend の Cloud Run は `--max-instances 1` で動かしており（ADR-003）、`--min-instances` を指定していないためアイドル時にインスタンスが破棄される。**コールドスタートは例外ではなく常態**であり、そのたびにキャッシュが空になって PokeAPI への再取得が走る。図鑑一覧は捕獲済みポケモンを直列に 1 匹ずつ取得するため、キャッシュが空だと捕獲数の 2 倍のリクエストが直列に飛ぶ。

一方で対象データは Gen 1-8 の 898 件に固定され、変化しない（ADR-004）。TTL も無効化も要らないデータを、実行時に取りに行く理由がない。

画像は `raw.githubusercontent.com` への直リンクで表示している。GitHub raw は CDN ではなく、未認証リクエストに IP ベースのレート制限をかけ、`Cache-Control` も 300 秒と短い。本番のアセット配信に使う経路として適切でない。

## 制約

外部コンテンツの扱いは BDR-007 に従う。ポケモン社の著作物を公開の場に再配布せず、キャッシュを非公開の範囲に留める。

backend は frontend との後方互換を保つ（ADR-016）。レスポンスからのフィールド削除は複数リリースに分ける必要がある。

## 詳細

### スナップショットの生成

`scripts/` に生成スクリプトを置き、明示的に実行したときだけ走らせる。CI・ビルド・デプロイからは呼ばない。実行のたびに一括取得が起きれば、スナップショットとして持つ意味を失うため。再実行するのは取得範囲を広げるときと、含める項目を増やすときに限られる。

取得元は `PokeAPI/api-data` の `data/api/v2/` 配下にある静的 JSON で、pokeapi.co が実際に配信しているデータそのものである。使うのは `pokemon-species/{1..898}` と `pokemon/{1..898}` の 2 種類。

出力は `domain` の `Pokemon` 型に変換済みの JSON とする。EN/JA の説明文ペアが得られない図鑑番号があれば生成時に失敗させる。現在は実行時に例外を投げているが、データの欠損は生成時に検出する。

全 898 件で生成を試したところ、1.85 MB（非圧縮）で、EN/JA ペアの欠損はなかった。

### 起動時にすべてメモリへ載せる

`PokemonClient` ポート（ADR-001）はそのままに、`SnapshotPokemonClient` を追加して `main.ts` の real モードで差し替える。`PokeAPIClient` は削除する。

起動時に Cloud Storage から JSON を読み、図鑑番号をキーとする `Map` と、タイプ別の図鑑番号の逆引きインデックスを構築する。以降の参照はすべてメモリから返るので、図鑑一覧の直列取得が消え、実行時のキャッシュミスがなくなる。パースとインデックス構築は実測 10 ミリ秒で、起動時のコストはほぼ 1.85 MB のダウンロードが占める。

タイプ別インデックスはスナップショットの `types` から構築できる。現在使っている `/type` エンドポイントの結果と全 18 タイプで一致することを確認済みで、生成スクリプトが `type` を取得する必要はない。

読み込み元は設定値で切り替える。ローカル開発の real モードでは Git 管理外のローカルパスから読む。mock モードは `MockPokemonClient` を使うため影響を受けない。

### 対象ポケモンの範囲を決める SSoT

どのポケモンが存在するかは、スナップショットの内容だけで決まる。`MAX_POKEMON_ID` は生成スクリプトが取得範囲を決めるためだけに使い、実行時の判定には使わない。実行時に両方を参照すると、環境変数とスナップショットの内容が食い違ったときに矛盾する。

### 画像の配信元と URL の作り方

`sprite_url` は backend のレスポンスに残し、値の作り方だけを変える。スナップショットには保存せず、図鑑番号と設定値のベース URL から実行時に組み立てる。フィールドを削って frontend に組み立てさせると契約の破壊的変更になり、複数リリースに分ける必要が生じる（ADR-016）。値の変更は後方互換なので、一度のリリースで完結する。

配信元は jsDelivr とする。GitHub の公開リポジトリ向け CDN で、同じバイト列を `Cache-Control: max-age=604800`（7 日）で配信する。GitHub raw の 300 秒より大幅に長い。参照はブランチではなくタグかコミットに固定する。

raw への直リンクは 3 か所にあり、すべて移す。

- backend が返すポケモンの `sprite_url`
- `ballAssets.ts` のボール画像
- `tutorialFixtures.ts` のチュートリアル用アートワーク

## 不採用案

### スナップショットを公開リポジトリにコミットする

898 件の図鑑説明文を公開リポジトリに置くことは、ポケモン社の著作物をデータセットとして再配布する行為にあたる（BDR-007）。

### ビルド時に Docker イメージへ焼き込む

起動時のダウンロードがなくなり実行時の外部依存も消えるが、CI に Cloud Storage の読み取り権限とビルド手順が増える。起動時ロードのコストが十分小さいため採らない。

### Firestore を L2 とする遅延永続キャッシュ

初回アクセス時だけ PokeAPI から取得して Firestore に書き、以降は Firestore から読む案。遊んだぶんだけ溜まる性質はあるが、コールドスタートが常態である以上、起動のたびに図鑑一覧で捕獲数ぶんの read が走る。まだ誰も遭遇していないポケモンには PokeAPI を叩くため、実行時の外部依存も残る。不変な 898 件のために永続ストアと read 課金を恒久的に払う理由がない。

### 画像を Cloud Storage に複製して配信

BDR-007 の方針に反する。技術的にも、自前ホストは PokeAPI の負荷を減らさず、解決したい GitHub raw のレート制限は jsDelivr で複製せずに済む。

画像を非公開バケットに置く案は、データのスナップショットと違って成立しない。ブラウザが直接取得する必要があるため、backend でプロキシする（単一インスタンスで API リクエストと帯域を奪い合う）か、署名付き URL を発行する（期限があり、URL が変わるためブラウザキャッシュが効かない）ことになる。
