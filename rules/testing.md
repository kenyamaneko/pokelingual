# テストの書き方

pokelingual のテスト規約。`principles.md` から参照される。
言語ごとの書き方 (データ駆動の書式等) は `rules/lang/<lang>.md` を見る。

## 基本方針

テストは実装をなぞるのではなく仕様を確かめる。

## 単体テストと結合テストの切り分け

- 関数に固有のロジック (計算式・分類・変換のルール) があるなら、期待値を具体値で書いて単体で確かめてよい。これは実装なぞりではない。
- ロジックを持たない関数に対しては単体テストを行わず、固有のロジックを持つ関数まで呼び出し元を遡り、結合テストの一環としてカバレッジを通せれば良い。

## 大原則

- 正常系だけでなく異常系も確かめる。依存先が失敗する・不正な入力が来るなど、うまくいかない経路も通し、その失敗が正しく扱われ呼び出し元まで伝わることを確かめる。
  - なぜ: 正常系だけでは、失敗時の処理が壊れていても気づけないため。とくに DB や外部 API をモックに差し替えるときは成功する場合だけを試しがちなので、失敗を返すケースも用意する。
- 外から観測できる結果だけをアサーションの対象にする。返り値・保存された値・応答・送られたメッセージ・画面表示など。どの関数が呼ばれたか、内部変数の値、どの分岐を通ったかといった実装の中身は検証しない。
  - なぜ: 実装の中身はリファクタで変わるため、縛ると振る舞いを変えていなくてもテストが壊れるため。
- 人間がテストの網羅性を確認しやすいよう、テスト名と中身を一致させる。テスト名と説明は仕様の言葉で書く。

## ロジックのテスト

- バグは境界に出やすいため、境界値を確かめる。条件の境界をまたぐ両側の値を入れる (例: 整数で a>0 なら 0 と 1)。
- 件数や有無で扱いが変わるなら、0 件・1 件・複数件、「無い」と「空」をケースに分ける。
- テストコードに if 文を書かず、条件ごとに独立したテストケースとして並べる。
  - なぜ: if があると、どのケースが実際に走ったか見えにくく、網羅できているか分からなくなるため(条件が不成立のとき何も確かめないまま通ることがある)。
- 実データの仕様変更でテストが壊れるのを防ぐため、テストでは本番の実データ ID をそのまま使わず、テスト専用のダミー値を用いる。

## 結合テスト
- 結合を確かめるテストは、途中の状態を手で用意して割り込ませず、公開された入口 (API・画面操作など)から実行する。
  - なぜ: 途中から始めると、入口から結果までの経路が壊れていても気づけないため。
- 操作した結果まで確かめる。「例外が出ない」「エラーが無い」「件数が合う」だけで終えず、操作の後に実際にどうなったか (保存された内容・返ってきた値・変化した状態) まで踏み込む。

## GUI (画面) のテスト

- 画面に見える結果で判断する。画面の切り替わり・表示内容・エラー表示を確かめ、モックがどう
  呼ばれたか・内部の値は検証しない。
- モックにするのは外部との境界だけ。通信・WebSocket・ログインなどを代用に差し替え、画面を
  組み立てる内部の部品は本物を使う。
  - なぜ: 部品どうしの繋がりが壊れていても気づけるようにするため。
- 見た目や実装の詳細に依存させない。CSS クラス名・アニメーションの文字列・配置の係数・固定の
  待ち時間などを確かめの対象にしない。
  - なぜ: 見た目を変えただけで壊れ、ユーザーに見える動きを確かめていないため。状態を見分けたいときは意味のある属性 (role / aria-* / data-state) で確かめる。
- 操作は実際のユーザー操作として発火させる (Testing Library 等)。

## アンチパターン

- 入力と出力の対応を、変換関数を直接呼んで突き合わせてはいけない。
  - 例: エラー→HTTP ステータスの変換関数を単体で呼び、対応を 1 件ずつ確かめる。コードの分岐を書き写しただけで、その状況で本当にその出力が返るかは確かめていない。
- 生成物の文字列を期待文字列とそのまま突き合わせてはいけない。
  - 例: 生成した SQL やコードを文字列比較で確かめる。書式を変えただけで壊れやすく、実行した結果が正しいかは確かめていない。
- 依存を欠けた状態で渡し、早期エラーだけを確かめてはいけない。
  - 例: ハンドラに依存を欠けた状態(nil/null/未設定など) で渡し、入力エラーの 400 や「落ちない」ことだけを確かめる。本来の処理(依存を呼んで結果を返す) に届かず、配線の欠落を見逃す。
- 実装をテストに書き写してはいけない。
  - 例: フィールドのコピーや定数をそのまま写す、期待値をテスト対象と同じ計算で作る。実装とテストが二重管理になり、両方に同じバグが入れば気づけない。
  - 除外: 純粋な関数 (計算式・分類・命名規則) を具体値で直接確かめる場合。
- 内部の関数を直接呼んではいけない。公開された経路から到達できる内部関数を、単体で直接呼んで確かめる。本来の使われ方を通っていない。
  - 除外: 公開経路から決して到達しない防御的な分岐 (default→例外 等) を確かめる場合。
- 本番に無い検証ロジックをテスト内で作って確かめてはいけない。
  - 例: 本番に存在しないロジックや変換をテスト内に書き、自前の入力に当てる。テスト内で完結し、本番コードを何も通っていない。
- テスト対象でないライブラリそのものの挙動を確かめてはいけない。ライブラリの品質はライブラリ開発者の責任であり、このプロジェクトで検証しても仕方がない。
