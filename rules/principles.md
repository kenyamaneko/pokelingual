## [base] 設計思想

- エラーは握りつぶさず根本解決する。ログだけして握りつぶすのは禁止
- デフォルト値へのフォールバックを行わない。意図しない値になるならエラーにする
- 一つの関数に複数の責務を負わせない

## [base] コーディング方針

- マジックナンバーを使用しない。定数化し、数値の用途がわかるようにする
- 関数などには docs コメントを付与する
  - 要約のみを書く。実装手順や設計判断の列挙はしない
  - 設計レベルの Why は docs/architecture.md に、行レベルの Why はその行のインラインコメントに分離する
- 実装との二重管理になるので What コメントを書いてはいけない
  - Why がかかる先を示すために、処理内容を要約の粒度で添えるのは可
- 実装から読み取れない意図は Why コメントとして記述する。設計レベルの意図なら docs/architecture.md に記述する
- backend↔frontend の契約型はリテラルで二重定義せず、SSoT である `shared/api-types` を参照する
- 未実装の TODO をコミットに残さない。残す場合は Issue を立ててリンクする
- 関数名は動詞から始め、何をする関数かを明確にすること
- 今使わないコードは残さない (動線があってもロジック的に呼ばれないコードを含む)
- 実装を変更したら関連 docs を同期する (ただし過去の ADR は変更しない)

## [base] 命名規約
- 関数・メソッドは名前から振る舞いが読み取れるようにする (原則として動詞で始める)
- 真偽を表す関数・変数は、真偽が読み取れる接頭辞を付ける (is / has / can / should 等)
- 一般的でない略語を使わない (ドメインで定着した語・各言語の確立した短縮語は可)
- 上記の具体化と言語固有の命名慣用は rules/lang/<lang>.md に従う。慣用が本節と衝突する場合は lang を優先する

## [base] ログ方針

- 構造化されたログを出力する
- ログ出力すべき事象
  - 呼び出し元に伝搬させられないエラーの補足
  - エラーではないが、その場所でしか気づけない事象 (未対応のイベント種別など)
- ログレベル
  - `Info`: 起動・停止・正常系の事象
  - `Warn`: システムの運用に影響を与えない、あるいは影響が軽微な事象
  - `Error`: システムの運用に支障をきたす事象

## [base] テスト方針

テストの書き方は @testing.md に従う。

## [base] サービス境界

- backend が Firestore への唯一のアクセス経路であり、自身が所有するデータの SSoT。frontend や外部プロセスは DB へ直接アクセスせず、backend の API 経由で行う

## [base] 実装フロー

- 開発は GitHub Issue を起点に行う
- 実装を変更したら、単体テストと統合テストが通過することを確認する

## [base] CI方針

- コスト削減のため、新規 GitHub Actions workflow を追加するときは `paths-ignore` / `timeout-minutes` / `concurrency` を必ず設定する
- 可読性を高めるため、ジョブの各ステップには `name` を付与する

## [base] 指示遵守

- ユーザーが明示した手段 (並列実行 / 特定ツール / 特定パス / 特定手順) は要件として扱い、希望と解釈しない
- 困難に直面しても、診断せず代替手段に切り替えない (= 「とにかく進める」を禁止)
- 障害発生時の最初の行動は根本原因の診断。迂回判断は原因を共有してから相談する

## [base] 判断権はユーザに属する

判断はユーザが持つ。Claude は実行者であって判断者ではない。

- 「Claude が判断してユーザが楽になる」と感じた瞬間が最も危険な合図。それは思い上がりであり、隠れた判断は後から発覚したときに節約したはずの労力を上回る負担をユーザに返す
- ユーザに見えない場所で結論を確定させない。判断材料を出し、判定はユーザに残す
- ルール・指示・仕様を softening / 簡略化 / 例外扱いすることは全て判断行為。揺らいだら必ずユーザに上げる
- audit / 報告 / 列挙で合否・軽重・優先度を Claude が pre-categorize しない

## [base] ルール違反への対応

- ルール違反がある状態でコミットしない。検出された違反は Claude が自律的に解消してからコミットする
- 違反を許容してよいかの判断をユーザに求めない。正当な逸脱理由がない単なる違反は、問わずに修正する
- ルールから外れる方が良いと判断した場合に限り、その理由を明記した上でユーザに判断を仰ぐ

## [base] 禁止事項

- Google Cloud のことを GCP と表現してはいけない（GCP は旧名称のため）
- `rules/` 配下のファイルを Claude が書き換えない (人間が運用するため、提案までに留める)
- 保守性を高めるため、原則としてエラー回避のためのワークアラウンドを書かない
